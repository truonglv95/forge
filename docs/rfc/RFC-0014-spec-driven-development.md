### 3.2. Spec schema

```yaml
---
spec_id: 2026-07-inline-completion
title: "Inline Tab Completion v1"
type: feature  # feature | bugfix | refactor | spike
status: draft  # draft | approved | implementing | done | rejected
author: truonglv95
created: 2026-07-24
updated: 2026-07-24
approver: ""
implementation_pr: ""
related_rfc: "RFC-0013"
---

## Context
Why this spec exists, what problem it solves.

## Requirements
- R1: Inline completion must trigger after 150ms debounce.
- R2: p50 latency < 500ms.
- R3: Tab to accept, Esc to dismiss.

## Design
High-level design, no implementation detail.

## Acceptance
- [ ] R1 verified via eval
- [ ] R2 verified via eval
- [ ] R3 manual test pass

## Out of scope
- Completion with context (Phase 2)
- FIM format support (Phase 2)
```

### 3.5. Validation hooks

`forge check` tб»ұ chбәЎy:
1. `forge spec validate` (tбәҘt cбәЈ specs)
2. Check: mб»—i spec `implementing` phбәЈi cГі `implementation_pr`
3. Check: mб»—i spec `done` phбәЈi cГі tбәҘt cбәЈ acceptance checkboxes ticked
4. Check: spec `approved` chЖ°a modify trong 7 ngГ y вҶ’ warn "stale approved"

```bash

вң“ All specs valid
вң— Spec 2026-07-inline-completion has status "implementing" but no implementation_pr
вң— Spec 2026-06-old-feature has been "approved" for 14 days without implementation
```

### 3.6. Agent hooks

Khi `forge ask` hoбә·c `forge agent run` Д‘Ж°б»Јc gб»Қi:

1. Agent tб»ұ scan `specs/` cho specs cГі status `approved` hoбә·c `implementing`.
2. Nбәҝu task intent match spec title/requirements, agent include spec content
   vГ o context manifest (source: `spec`).
3. Agent prompt include: "This task may relate to spec <id>. Read the spec
   requirements before proposing."
4. Khi agent propose, ghi `spec_id` vГ o proposal metadata.
5. `forge spec trace <id>` query `.forge/traces/<id>.jsonl` cho tбәҘt cбәЈ
   proposals + transactions + runs liГӘn quan.

## 6. Rollout plan

1. **Week 1:** `forge spec init` + directory convention + templates. `spec
   validate` basic.
2. **Week 1-2:** Spec metadata `.forge/specs/<id>.json` + `spec link`.
3. **Week 2-3:** `forge check` validation hooks.
4. **Week 3:** Agent hooks: spec scan + context include + proposal metadata.
5. **Week 3-4:** `forge spec trace` JSONL accumulation.
6. **Week 4-5:** IDE: spec list panel, spec editor, trace view.

## 7. Risks

| Rб»§i ro | GiбәЈm |
|---|---|
| Spec rot (specs khГҙng update) | `forge check` warn stale |
| Spec too prescriptive (block agent creativity) | Spec lГ  requirements, khГҙng phбәЈi implementation |
| Directory convention xung Д‘б»ҷt | Support cбәЈ `specs/` vГ  `.kiro/specs/` |
| Performance (scan specs mб»—i run) | Cache spec index trong `.forge/specs/index.json` |
| Adoption resistance | Templates + `forge spec init` reduce friction |

## 8. Alternatives considered

- **GitHub Issues as specs:** вқҢ KhГҙng cГі structured requirements + acceptance.
- **Jira/Linear integration:** вқҢ Out of scope, Forge lГ  local-first.
- **Specs in code comments:** вқҢ KhГҙng cГі separate lifecycle.
